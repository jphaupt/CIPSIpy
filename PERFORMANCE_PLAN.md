# CIPSI Performance Plan (AI-Generated)

> Disclaimer: This report was generated by AI (GitHub Copilot, GPT-5.3-Codex) from the current codebase state and test behavior. It is a technical proposal, not a formally benchmarked performance study.

## Scope

This plan focuses on improving runtime of the current CIPSI implementation while preserving correctness first.

Current correctness target in tests:
- Full CIPSI-vs-FCI checks: `H2/sto-3g`, `HeH+/sto-3g`
- One medium smoke test: `H3+/3-21g` with `max_dets=5`

## What Was Fixed for Correctness

1. Variational-space growth loop in `run_cipsi` was completed.
2. Selected external determinants are now added to the internal variational set and re-diagonalized.
3. Single excitations are now included exactly once through selective untagging in `apply_epv_and_single_tagging`.

This fixed the HeH+ mismatch (previously missing determinants from single-excitation channels).

## Why Runtime Blows Up

The main hotspot is `CIPSISolver.run_unfiltered_selection` in `src/cipsipy/cipsi.py`.

The current structure is deeply nested Python loops over:
- `(p, q)` annihilation pairs in spin-orbitals
- generator determinants
- selector determinants
- `(r, s)` creation pairs

Pseudo-structure (from current code):

```python
for ps in range(2*self.ham.norb):
    for qs in range(ps+1, 2*self.ham.norb):
        for i_gen in range(N_gen):
            ...
            for j_sel in range(N_sel):
                excited_pairs = get_creation_pairs(...)
                for rs, ss in excited_pairs:
                    Pmat_pq = Pmat_pq.at[rs, ss].add(...)
            for rs in range(2*self.ham.norb):
                for ss in range(rs + 1, 2*self.ham.norb):
                    ...
```

Asymptotic cost is roughly:

`O((2N)^2 * N_gen * N_sel * K_pairs + (2N)^4 * N_gen)`

with substantial constant factors from Python control flow.

## Why LiH 6-31G* Feels "Stuck" Before Iter 1

The apparent stall is expected with the current implementation: the expensive
selection work happens between iteration prints.

`run_cipsi` prints once per completed macro-iteration. In each iteration, the
print occurs after:
1. diagonalization
2. full call to `run_unfiltered_selection`
3. PT2 aggregation

So if selection is slow, there is no intermediate progress output.

For LiH/6-31gstar in this repo:
- `norb = 15` spatial orbitals
- spin-orbital count is `2*norb = 30`
- annihilation pair count is `C(30, 2) = 435`
- inner `(r, s)` pair count is also `C(30, 2) = 435`

Compared with H3+/3-21g (`norb=6`, `2*norb=12`, `C(12,2)=66`), the pair-space
blowup is:
- `435 / 66 ~= 6.6x` larger in outer `(p,q)` batches
- another `6.6x` in inner `(r,s)` sweeps
- roughly `~44x` more pair-combinations even before considering larger
    `N_gen`/`N_sel` and Python overhead

Practical consequence for one macro-iteration:
- H3+ may finish quickly enough to print every few seconds
- LiH spends most wall time inside the nested Python loops in
    `run_unfiltered_selection` before the next `iter=...` line appears

Important: this is not primarily Davidson cost. In early iterations, Davidson
works on a tiny variational space and is cheap relative to selection.

The dominant costs are:
- repeated Python loops over `(p,q,i_gen,j_sel,r,s)`
- repeated scalar `ham.element(...)` evaluations
- many `jnp.ndarray.at[...]` updates in eager mode (functional array copies)

This is why LiH can look like it is "hanging" even though it is still making
progress.

## Explicit Bottlenecks with Real Code Examples

### 1. Repeated JAX `.at[...]` inside Python loops

In `src/cipsipy/cipsi.py`, mask and matrix updates are done one element at a time:

```python
Pmat_pq = Pmat_pq.at[rs, ss].add(c_J * h_elem)
Pmat_pq = Pmat_pq.at[ss, rs].add(c_J * h_elem)
```

and:

```python
tagmask = tagmask.at[r_s_pair[0], r_s_pair[1]].set(False)
tagmask = tagmask.at[r_s_pair[1], r_s_pair[0]].set(False)
```

This incurs heavy overhead because each update creates a new array object in eager mode unless the entire loop is JIT-fused (which it currently is not).

### 2. Python-driven branching for each determinant pair

Examples in `run_unfiltered_selection`:

```python
if (not is_orbital_occupied(Gdet, ps)) or (not is_orbital_occupied(Gdet, qs)):
    continue
```

```python
if not tagmask[rs, ss]:
    continue
```

```python
if jnp.isclose(denom, 0.0):
    continue
```

These branches are correct, but they prevent vectorized throughput and dominate runtime for larger systems.

### 3. Recomputing expensive connectivity repeatedly

`get_creation_pairs(G_pq, Sdet, norb)` is called for every `(G_pq, Sdet)` combination. The same structural relationships are revisited many times across iterations.

### 4. Dense-sized temporary matrix per `(p, q, G)` batch

```python
Pmat_pq = jnp.zeros((2*self.ham.norb, 2*self.ham.norb))
```

This is reallocated frequently even though only a sparse subset of entries is usually used.

### 5. Hamiltonian diagonal and matvec are loop-heavy

In `src/cipsipy/hamiltonian.py`:

```python
for i in range(len(h_diag)):
    h_ii = _diagonal_element(...)
    h_diag = h_diag.at[i].set(h_ii)
```

and matvec computes connected pairs via Python loops and repeated scalar element evaluations.

## Near-Term Optimization Plan (Keep Algorithm, Improve Implementation)

### Phase 1: Low-risk improvements

1. Replace repeated scalar `.at` updates with batched indexed updates.
2. Precompute occupied-orbital masks for each determinant once per iteration.
3. Cache diagonal elements `H_aa` for external determinants in a dict during one selection pass.
4. Cache `get_creation_pairs` results for repeated `(G_pq, Sdet)` patterns within one iteration.

Expected impact: significant constant-factor reduction without changing mathematical behavior.

### Phase 2: Dataflow refactor for vectorization/JIT

1. Move selection kernel to pure-array style functions (minimal Python branches).
2. Use JAX vectorized operations (`vmap`/batched gather/scatter) over selector sets.
3. Build sparse-like accumulator buffers (`rs_idx`, `value`) and reduce once per batch.

Expected impact: large speedup on medium systems and better scaling with orbital count.

### Phase 3: Algorithmic reductions

1. Restrict candidate `(r, s)` generation using stronger connectivity prefilters.
2. Use determinant hash/index maps to skip duplicate generation early.
3. Optional semistochastic screening of tiny couplings for large spaces.

Expected impact: reduced effective complexity and memory traffic at large `N_det`.

## Spin-Pure Determinant Completion (Section 5.8)

Would adding spin-completion determinants help? Yes for spin contamination control, but it is not the root cause of the previously failing HeH+ test.

- HeH+ mismatch was fixed by single-excitation handling.
- Spin-pure completion is still a valuable next feature.

Suggested implementation point:
- After external determinant selection and before building next variational wavefunction in `run_cipsi`, augment selected determinants with spin-flip partners that preserve `N_alpha` and `N_beta` constraints.
- Deduplicate against current variational set.

Pseudo-hook:

```python
selected = self._select_external_determinants(contribs, selection_fraction)
selected = self._augment_spin_pure(selected)  # new step
```

Validation plan for spin-pure feature:
1. Add a nontrivial open-shell system test where spin contamination can appear.
2. Track `<S^2>` before and after augmentation.
3. Verify energy invariance/improvement and deterministic determinant counts.
